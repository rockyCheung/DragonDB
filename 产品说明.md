## 轻量级分布式数据库设计：DragonDB

### 1. 设计背景与目标
传统关系型数据库（RDBMS）以严格的模式（Schema）、ACID事务和SQL查询为核心，但在海量数据、高并发、水平扩展和灵活数据结构的场景下显得力不从心。为此，我们设计一款**轻量级分布式数据库——DragonDB**，旨在：
- **提供关系模型的替代方案**：放弃复杂的表连接和固定模式，采用更灵活的文档模型，简化数据关系表达。
- **实现轻量级部署**：单节点占用资源少（内存<100MB，磁盘依赖存储引擎），支持快速启动和简易运维。
- **具备分布式能力**：自动数据分片、多副本容错、可调一致性，支持跨机器水平扩展。
- **保持简单易用**：提供RESTful API和简易查询语法，降低使用门槛。

### 2. 总体架构
DragonDB采用**去中心化、对等节点**的架构，无单点故障。每个节点功能对等，通过Gossip协议维护集群成员和元数据。

- **节点角色**：所有节点平等，均可处理读写请求。
- **数据分区**：使用**一致性哈希**将数据分布到多个虚拟节点（vnode），每个vnode映射到物理节点。添加或移除节点时仅需迁移少量数据。
- **副本机制**：每个数据分区有N个副本（N可配置，默认3），副本分散在不同节点以提高容错。
- **一致性模型**：支持**可调一致性**（Quorum机制），读写操作可指定`r`（读所需节点数）和`w`（写所需节点数），实现从强到最终的一致性。
- **冲突处理**：采用**向量时钟**（Vector Clock）检测并发更新，发生冲突时由应用层或基于时间戳的“最后写入胜利”（LWW）策略解决。

### 3. 数据模型：文档模型替代关系模型
DragonDB使用**JSON文档**作为基本数据单元，摒弃了关系模型的表、行和严格模式。

- **集合（Collection）**：类似于关系库的表，但无固定模式。同一集合中的文档可以拥有不同的字段。
- **文档（Document）**：由键值对组成，支持嵌套对象和数组，使用唯一ID标识（可自动生成或指定）。
- **索引**：支持对文档中的字段创建二级索引（如B+树或LSM树索引），加速查询。
- **关系表达**：
  - **嵌入式文档**：将关联数据直接嵌套，例如用户文档中嵌入地址数组，避免单独建表。
  - **引用**：在文档中保存另一文档的ID，通过应用层二次查询或数据库提供的`$lookup`（类似join但非核心功能）实现关联，但鼓励反范式化以提升性能。

**对比关系模型**：DragonDB通过文档的自描述特性实现了**模式灵活性**，并鼓励将关联数据聚合存储，从而**消除昂贵的join操作**，更适合读写比例高、数据结构多变的场景。

### 4. 分布式核心设计
#### 4.1 数据分区与路由
- 一致性哈希环：每个文档的ID或指定分区键经过哈希后落在环上某个vnode，vnode所属节点负责该数据。
- 客户端驱动或节点转发：客户端可缓存路由表，直接请求目标节点；或发送请求到任意节点，由该节点代理转发。

#### 4.2 复制与一致性
- 写操作：客户端发送请求到分区的主节点（或任意节点，由协调者处理），协调者将写请求并行发送给所有副本节点，等待至少`w`个节点成功确认后返回客户端。
- 读操作：协调者向所有副本请求数据，根据向量时钟合并版本，返回最新数据给客户端；若读到的版本不一致且未达到最新，可触发读取修复。
- 故障检测：通过Gossip协议交换节点心跳，发现节点离线时自动将副本迁移至其他健康节点。

#### 4.3 轻量级实现
- **存储引擎**：嵌入RocksDB（或类似LSM-tree引擎），提供高效的读写和压缩能力，同时保持低内存占用。
- **网络层**：基于HTTP/1.1或gRPC，支持简单的RESTful API，减少协议开销。
- **依赖精简**：使用Go语言开发，静态编译，无外部依赖，便于跨平台部署。

### 5. API与查询语言
DragonDB提供简单直观的RESTful API和类MongoDB的查询语法（通过JSON表示）。

- **基本操作**：
  - `PUT /collections/{collection}/documents/{id}` 创建/更新文档
  - `GET /collections/{collection}/documents/{id}` 读取文档
  - `DELETE /collections/{collection}/documents/{id}` 删除文档
- **查询**：支持基于字段的过滤、范围查询、排序和分页。
  ```
  POST /collections/{collection}/query
  {
    "filter": {"age": {"$gt": 18}},
    "sort": {"name": 1},
    "limit": 10
  }
  ```
- **聚合**：提供简单的MapReduce框架或管道聚合（如`$group`、`$project`），但鼓励在应用层完成复杂处理以保持轻量。

### 6. 替代关系关系的实践策略
DragonDB通过以下方式替代传统的关系模型：
- **无模式设计**：字段可动态增减，适应快速迭代的业务，无需执行昂贵的ALTER TABLE。
- **聚合存储**：将相关数据保存在一个文档中，例如订单包含商品列表和收货地址，避免多表join。
- **引用与反范式**：对于多对多关系，可采用双向引用并配合应用逻辑维护一致性，或者冗余必要字段。
- **最终一致性**：允许短暂的数据不一致，换取高可用和低延迟，适用于社交媒体、用户画像等场景。

### 7. 适用场景与局限性
**适用场景**：
- IoT数据采集（设备信息、时间序列）
- 用户资料和配置管理
- 内容管理系统（文章、评论）
- 实时计数器、排行榜

**局限性**：
- 不支持跨文档事务（仅单文档原子性）
- 复杂关联查询需要应用层处理
- 强一致性场景需配置高读写quorum，可能降低性能

### 8. 总结
DragonDB作为一款轻量级分布式数据库，通过文档模型、一致性哈希和可调一致性，提供了关系型数据库之外的一种新选择。它在扩展性、灵活性和部署简易性上具有优势，特别适合中小规模、快速变化和读密集型的应用。未来可考虑增加更多查询优化、多租户支持和监控工具，进一步完善生态。
