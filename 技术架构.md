## 基于 Python 的轻量级分布式数据库 DragonDB 设计

### 一、设计目标
- **语言**：Python 3.10+，利用 `asyncio` 实现高并发 I/O。
- **数据模型**：文档型（JSON），支持动态模式，替代传统关系模型的表结构。
- **分布式能力**：
  - 自动数据分片（一致性哈希 + 虚拟节点）
  - 可配置副本数（默认 3）
  - 可调一致性（Quorum NWR）
- **轻量级**：单节点内存占用 < 200MB，可基于文件或内存存储，无外部依赖。
- **接口**：RESTful API（HTTP/JSON），兼容类 MongoDB 查询语法。

### 二、整体架构
DragonDB 由多个对等节点组成，每个节点独立运行，通过 Gossip 协议维护集群视图。所有节点均可处理读写请求，请求由协调者（Coordinator）负责路由和执行。

### 三、核心元组件（Python 类设计）

#### 1. `Node` – 节点抽象
每个节点包含以下子组件：
- `storage_engine`：管理数据的存储和索引
- `cluster_manager`：维护集群节点列表、分区映射表
- `coordinator`：处理客户端请求，执行 Quorum 逻辑
- `http_server`：基于 `aiohttp` 的 REST API 服务
- `gossip`：定期与其他节点交换心跳和元数据

#### 2. `StorageEngine` – 存储引擎
- 基于 `RocksDB` 或 Python 内置的 `shelve`（生产环境推荐 RocksDB，但演示可用内存字典 + 文件快照）。
- 数据组织：每个集合对应一个键空间，文档以 `{collection}:{doc_id}` 为键，值存储 JSON 序列化后的字节并附带版本向量。
- 支持二级索引：每个索引对应一个额外的 B+ 树结构（可用 `sortedcontainers` 库模拟）。
- 接口：
  ```python
  class StorageEngine:
      async def get(self, collection, doc_id) -> Document
      async def put(self, collection, doc_id, document, version_vector) -> bool
      async def delete(self, collection, doc_id) -> bool
      async def query(self, collection, filter_expr, limit, offset) -> List[Document]
  ```

#### 3. `Document` – 文档模型
```python
@dataclass
class Document:
    id: str                     # 文档唯一 ID
    collection: str
    data: dict                  # JSON 数据
    version_vector: Dict[str, int]   # 节点 ID -> 版本号，用于冲突检测
    timestamp: float             # 可选，用于 LWW
```

#### 4. `Partition` 与 `Replica` – 分片与副本
- 逻辑分区（Partition）由一致性哈希的虚拟节点构成，每个分区包含 R 个副本（Replica），每个副本存储在一个物理节点上。
- 分区映射表：`Dict[partition_id, List[node_id]]`，表示某个分区的所有副本所在节点。
- 在节点启动时，通过 `cluster_manager` 从种子节点同步映射表，或通过 Gossip 协议动态更新。

#### 5. `ClusterManager` – 集群元数据管理
- 维护：`nodes`（在线节点列表）、`partition_map`（分区 → 副本节点列表）、`node_partitions`（节点 → 负责的分区列表）
- 使用一致性哈希算法（引入虚拟节点）计算文档所属分区：
  ```python
  def get_partition(doc_id: str) -> int:
      hash_val = hash(doc_id)  # 使用一致性哈希环
      return partition_id
  ```
- 提供接口：`get_replicas(partition_id) -> List[node_id]` 获取某分区的所有副本节点。

#### 6. `Coordinator` – 读写协调器
协调器负责执行 Quorum 逻辑：
- **写操作**：
  1. 根据文档 ID 计算分区。
  2. 从 `cluster_manager` 获取该分区的所有副本节点地址。
  3. 向所有副本并发发送写请求，附带当前文档的版本向量（若更新需携带旧版本）。
  4. 等待至少 `w` 个节点成功响应，若成功则返回客户端；否则返回失败。
  5. 若版本冲突（向量时钟不可比），则返回冲突并由客户端解决或使用 LWW 策略。
- **读操作**：
  1. 同样获取副本节点列表。
  2. 向所有副本发送读请求，收集返回的文档及版本向量。
  3. 合并版本（取最新，若冲突则标记），等待至少 `r` 个节点响应。
  4. 若有版本冲突，可尝试读取修复（将最新版本写回过期的副本）。

#### 7. `Gossip` – 集群成员维护
- 每个节点周期性地随机选择几个节点交换状态（节点存活、分区负载等）。
- 使用 `pyee` 或自定义事件通知其他组件节点变动。

### 四、基础功能流程

#### 4.1 写请求 (PUT /collections/{coll}/documents/{id})
```
Client → Node A (Coordinator)
  │
  ├─ 计算 partition = hash(id)
  ├─ 获取 replicas = [node1, node2, node3]
  ├─ 并发发送写请求到所有 replicas（携带文档数据、版本向量）
  │   各 replica 执行本地存储：
  │     - 若文档不存在，直接写入并设置版本向量为 {node_id: 1}
  │     - 若存在，比较版本向量：
  │         如果新版本向量因果依赖旧版本，则更新并增加本地计数器；
  │         如果冲突，则保留两个版本（或根据 LWW 解决）
  │     - 返回成功/失败及最新版本向量
  ├─ 等待至少 w 个成功响应
  └─ 返回客户端 (201 Created / 409 Conflict)
```

#### 4.2 读请求 (GET /collections/{coll}/documents/{id})
```
Client → Node A (Coordinator)
  │
  ├─ 计算 partition = hash(id)
  ├─ 获取 replicas
  ├─ 并发发送读请求到所有 replicas（可指定读取级别，如 r=2）
  ├─ 各 replica 返回文档及版本向量
  ├─ 合并版本（选择最新或标记冲突）
  ├─ 等待至少 r 个响应
  ├─ 若有版本冲突，返回合并后的多个版本（或根据 LWW 选择）
  └─ 返回文档给客户端
```

#### 4.3 节点加入与分区再平衡（简化版）
- 新节点启动后向种子节点注册，种子节点通过 Gossip 广播新节点信息。
- 触发分区映射重新计算：将部分虚拟节点从现有节点迁移到新节点。
- 迁移过程：源节点将对应分区的数据复制到目标节点，完成后更新元数据。
- 简化实现：初次版本可支持静态配置，不实现自动再平衡，仅手动触发。

### 五、关键算法

#### 5.1 一致性哈希（带虚拟节点）
- 每个物理节点生成 100~200 个虚拟节点，均匀分布在哈希环上。
- 使用 `sortedcontainers.SortedDict` 存储哈希值到虚拟节点的映射。
- 文档 ID 的哈希值在环上找到第一个大于它的虚拟节点，该虚拟节点所属物理节点即为主副本；后续虚拟节点负责其他副本。

#### 5.2 Quorum NWR
- 可配置参数：`N` = 副本数，`W` = 写成功所需节点数，`R` = 读成功所需节点数。
- 满足 `W + R > N` 时可实现强一致性（读取总能看到最新写入）。
- 实现时，协调器在请求头中附带 `w` 和 `r` 参数，覆盖默认值。

#### 5.3 向量时钟冲突解决
- 每个节点维护一个单调递增的计数器，作为该节点的版本号。
- 写入时，如果文档已存在，协调器将现有版本向量与本地版本向量合并（每个维度取 max），然后增加协调器所在节点的计数器。
- 若协调器收到的版本向量与本地版本存在“并发”关系（即不可比），则标记冲突，返回多个版本给客户端，由客户端决定如何合并（或使用 LWW 自动选择最新时间戳）。

### 六、存储实现细节
- **文档存储**：以 `collection/doc_id` 为键，值存储 JSON 序列化后的字节，以及版本向量（可序列化后附加在值后或单独存储）。
- **二级索引**：使用倒排索引，每个索引字段对应一个 B+ 树，键为字段值，值为文档 ID 列表。查询时先通过索引缩小范围，再读取文档。
- **持久化**：可定时将内存数据写入磁盘快照（snapshot）和预写日志（WAL），保证重启恢复。

### 七、API 设计示例（RESTful）

#### 集合操作
- `PUT /collections/{collection}` – 创建集合（可选，也可隐式创建）
- `DELETE /collections/{collection}` – 删除集合

#### 文档操作
- `PUT /collections/{collection}/documents/{id}` – 创建/更新文档（需在 body 中提供 JSON）
- `GET /collections/{collection}/documents/{id}?r=2` – 读取文档
- `DELETE /collections/{collection}/documents/{id}` – 删除文档
- `POST /collections/{collection}/query` – 查询
  ```json
  {
    "filter": {"age": {"$gt": 25}, "city": "Beijing"},
    "sort": {"name": 1},
    "limit": 10,
    "offset": 0
  }
  ```

#### 节点管理（仅内部或管理员）
- `GET /cluster/nodes` – 列出所有节点
- `POST /cluster/join` – 新节点加入集群（指定种子节点）

### 九、轻量级特性体现
- **无外部依赖**：仅使用 Python 标准库 + `aiohttp`（HTTP 服务） + `sortedcontainers`（一致性哈希环）即可运行。
- **可配置存储**：默认使用内存存储 + 文件快照，适合开发测试；生产可替换为 RocksDB（需安装 `python-rocksdb`）。
- **单节点模式**：支持以单节点运行，此时不启动 Gossip，分区映射仅包含自己，副本数=1。
- **易于扩展**：新增功能如事务、监控可通过插件方式添加。

### 十、总结
本设计方案展示了如何用 Python 实现一个具备分布式能力的文档数据库 DragonDB。核心元组件（节点、存储引擎、集群管理器、协调器）各司其职，通过一致性哈希和 Quorum 机制实现了数据分片与复制，同时使用向量时钟处理版本冲突。RESTful API 使得接入简单，文档模型替代了关系模型的复杂性，适用于快速变化的数据场景。该设计虽简化了部分实现（如自动再平衡、故障恢复），但提供了一个可运行的基础框架，可根据实际需求进一步扩展。
