## 自研持久化存储引擎设计：DragonStore

### 1. 设计背景与目标
借鉴 RocksDB 的 LSM 树思想，设计一个轻量级、可嵌入的持久化存储引擎 **DragonStore**，用于 DragonDB 的本地存储。目标如下：
- **减少外部依赖**：纯 Python 实现（或核心 C 扩展，但保持 API 简单），不依赖 RocksDB 等第三方库。
- **高性能读写**：利用 LSM 结构优化写入，通过缓存和索引加速读取。
- **扩展灵活**：支持可插拔的合并策略、压缩算法、索引格式，便于未来定制。
- **崩溃一致**：通过 WAL 和原子操作保证数据不丢失。

### 2. 总体架构
DragonStore 采用分层设计，核心组件包括：

```
┌─────────────────────────────────────┐
│            DragonStore API          │
├─────────────────────────────────────┤
│  ┌─────────┐  ┌─────────┐  ┌─────┐  │
│  │MemTable │  │  Cache  │  │ ... │  │
│  └─────────┘  └─────────┘  └─────┘  │
├─────────────────────────────────────┤
│         Compaction Manager           │
├─────────────────────────────────────┤
│    WAL       │   SSTable Files       │
└──────────────┴───────────────────────┘
```

- **MemTable**：内存中的有序数据结构（跳表），暂存最新写入。
- **WAL（预写日志）**：追加日志文件，用于崩溃恢复。
- **SSTable（有序字符串表）**：磁盘上的只读文件，存储有序键值对。
- **Compaction Manager**：后台线程，合并 SSTable 文件，清理过期数据。
- **Cache**：缓存 SSTable 的数据块，加速读操作。
- **Bloom Filter**：可选，每个 SSTable 附带，快速判断键是否不存在。

### 3. 核心组件设计

#### 3.1 MemTable
- **数据结构**：采用跳表（Skip List），支持 O(log n) 的插入和查询。
- **内存限制**：当 MemTable 大小达到阈值（如 4MB）时，转为不可变 MemTable，并触发生成 SSTable。
- **并发控制**：使用读写锁（如 `asyncio.Lock` 或线程锁）保证线程安全。

#### 3.2 WAL（预写日志）
- **格式**：二进制追加日志，每条记录包含：键长度、键数据、值长度、值数据、操作类型（Put/Delete）。
- **刷盘策略**：可配置每次写入强制刷盘（`sync`）或定期刷盘（默认每秒），平衡性能与可靠性。
- **日志切换**：当 MemTable 持久化为 SSTable 后，对应的 WAL 段可被删除。

#### 3.3 SSTable
- **文件格式**：分为数据块和索引块。
  - **数据块**：存储连续的键值对，默认块大小 4KB。
  - **索引块**：存储每个数据块的最后一个键及其偏移量，用于二分查找定位数据块。
  - **元数据**：文件末尾存储 Bloom Filter 数据、索引位置、版本信息等。
- **命名规则**：`{level}_{sequence}.sst`，如 `L1_0001.sst`，层级从 0 开始。

#### 3.4 Compaction Manager
- **触发条件**：当某层 SSTable 数量或大小超过阈值时触发合并。
- **合并策略**：借鉴 Leveled Compaction（分层合并）或 Size-Tiered Compaction（大小分级），可配置。
  - **Level 0**：由 MemTable 直接生成的 SSTable，文件可能重叠。
  - **Level 1+**：合并后文件有序且不重叠。
- **合并过程**：读取多个输入 SSTable，归并排序输出新的 SSTable，同时清理被覆盖或删除的键。

#### 3.5 Cache
- **Block Cache**：缓存最近使用的数据块，使用 LRU 策略。
- **实现**：可用 `collections.OrderedDict` 或第三方库 `cachetools`。

#### 3.6 Bloom Filter
- **用途**：加速点查询，避免无效的 SSTable 读取。
- **实现**：每个 SSTable 生成一个 Bloom Filter（如使用 `pybloom_live` 或自研位数组），存储所有键的哈希。

### 4. 读写流程

#### 4.1 写操作
1. 将操作（Put/Delete）追加到 WAL（Write Ahead Log），确保持久化。
2. 更新 MemTable（跳表插入/删除）。
3. 若 MemTable 大小超限，将其转为不可变 MemTable，并启动后台任务将内容写入 SSTable（Level 0）。
4. 写入完成后，通知 Compaction Manager 可能触发合并。

#### 4.2 读操作
1. 先在 MemTable 中查找，若找到直接返回（注意删除标记）。
2. 若未找到，在不可变 MemTable 中查找（如果有）。
3. 若仍未找到，从 Level 0 开始逐层查找 SSTable：
   - 对每一层，先检查 Bloom Filter，若不存在则跳过该 SSTable。
   - 若可能存在，加载 SSTable 的索引，二分定位数据块，从 Block Cache 或磁盘读取数据块，查找目标键。
4. 返回找到的值或 `None`。

### 5. 合并策略详解（Leveled Compaction）
- **Level 0**：由 MemTable 直接落盘的文件，文件之间可能存在重叠区间。
- **Level 1+**：每个层级文件总大小呈倍数增长（如 10 倍），且文件间不重叠。
- **触发条件**：
  - Level 0 文件数超过阈值（如 4）时，选取一部分文件合并到 Level 1。
  - 其他层级文件总大小超过阈值时，选取一个文件与下一层重叠区间合并。
- **合并过程**：采用多路归并，生成新的 SSTable，删除旧文件。

### 6. 崩溃恢复
- 启动时，扫描 WAL 文件，重放所有操作重建 MemTable。
- 若 WAL 对应的 MemTable 已持久化（存在对应的 SSTable），则跳过该 WAL 段。
- 恢复完成后，将未刷盘的 MemTable 转为 SSTable。

### 7. 扩展灵活性设计
DragonStore 通过接口和插件机制支持灵活扩展：

- **合并策略可插拔**：定义 `CompactionPolicy` 接口，用户可实现自定义合并策略（如 Time-Tiered、FIFO）。
- **压缩算法可替换**：每个 SSTable 数据块可配置压缩算法（如 Snappy、Zlib、LZ4），通过工厂模式创建压缩器。
- **索引格式可定制**：SSTable 的索引可以是稀疏索引、二分查找索引，甚至支持前缀索引，由用户选择。
- **Bloom Filter 实现可替换**：可选用不同哈希函数或布谷鸟过滤器。
- **缓存策略可配置**：LRU、LFU 或 TinyLFU 等。

这种设计使得 DragonStore 既能保持核心 LSM 结构的优势，又能根据应用场景调整内部组件，满足 DragonDB 未来多样化的需求。

### 8. 与 DragonDB 的集成
DragonStore 提供简单的键值接口（`put(key, value)`、`get(key)`、`delete(key)`、`write_batch(operations)`），上层存储引擎（如 `RocksDBStorage` 的替代）将其包装为文档存储：

- **键编码**：将 `collection:doc_id` 编码为字节。
- **值编码**：序列化文档（含版本向量）为字节。
- **索引管理**：通过 WriteBatch 原子更新文档和索引条目（索引可单独存储在 DragonStore 中，以 `idx:{field}:{value}:{doc_id}` 为键）。

### 9. 实现考量
- **语言选择**：Python 实现可快速迭代，但性能可能受限。关键组件（如 MemTable 跳表、Bloom Filter）可用 C 扩展优化。
- **异步支持**：DragonDB 基于 `asyncio`，因此存储引擎需支持异步 I/O（如使用 `aiofiles` 或线程池包装同步操作）。
- **文件管理**：需谨慎处理文件锁、目录结构，避免多线程竞争。

### 10. 总结
DragonStore 借鉴 RocksDB 的 LSM 树思想，通过自研组件实现了可嵌入、可扩展的持久化存储引擎。其核心设计包括 MemTable、WAL、SSTable、合并管理器、缓存和 Bloom Filter，并提供插件机制支持灵活定制。与 DragonDB 集成后，既能减少外部依赖，又能保证存储层的性能与可靠性，为 DragonDB 的分布式能力奠定坚实基础。
